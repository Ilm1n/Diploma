## Почему диаграммы разделены

Одна большая use-case диаграмма для текущего проекта получается слишком перегруженной:

- много ролей и пересечений прав
- много сценариев внутри доски
- приглашения и роли быстро делают схему трудночитаемой

Поэтому используется связка:

- 1 обзорная диаграмма верхнего уровня
- 2 детальные диаграммы по ключевым логическим блокам

## Что сознательно не включено в use-case диаграммы

Use-case диаграмма должна показывать цели внешнего актора, а не внутреннюю реализацию системы. Поэтому в диаграммы не включаются как самостоятельные use case:

- `restoreSession`
- refresh-token flow
- WebSocket heartbeat
- client mutation id
- внутренняя синхронизация состояния store
- технические шаги загрузки данных и повторных запросов

Эти механизмы можно упомянуть в тексте пояснения к диаграммам или показать на sequence/activity диаграммах, если это понадобится на защите.

## Как трактовать права

При построении use-case диаграмм за основу взяты backend-ограничения доступа, а не только видимость кнопок во frontend.

Это важно, потому что часть UI-сценариев визуально доступна шире, чем реально разрешено API. Для защиты корректнее опираться именно на серверную авторизацию.

Ключевые ограничения:

- `OWNER` обновляет и удаляет проект, а также меняет роли участников
- `MANAGER` управляет приглашениями, участниками, колонками, тегами и любыми задачами
- `MEMBER` может читать проект и доску, создавать задачи и изменять только задачу, назначенную на него

## Если понадобится ещё одна диаграмма

При необходимости можно вынести отдельную детальную диаграмму:

- `03-auth-profile.puml` — регистрация, вход, выход, профиль, аватар, путь пользователя по приглашению

Пока этот блок оставлен на общей диаграмме, чтобы не дробить материал сильнее, чем нужно для защиты.
